## 问题概述
AiNiee 在翻译 Word 文档时，将文档按照 XML 中的 w:t 节点切分成翻译单元。由于切分过于细碎，导致：
- 上下文丢失
- 翻译质量下降
- 翻译成本增加

## 改进方案

### 方案 1：段落级合并（推荐）✅

**核心思路**：以段落为单位进行翻译，而不是以 w:t 节点为单位

**优点**：
- ✅ 完全保留上下文
- ✅ 翻译质量最高
- ✅ 实现相对简单
- ✅ 处理所有格式差异（字体、上下标、颜色等）

**缺点**：
- ⚠️ 长段落可能导致 token 超限
- ⚠️ 回写时需要整段替换

**实现要点**：
1. 以 w:p（段落）为单位提取完整文本
2. 将完整段落发送给翻译模型
3. 回写时替换整个段落内容
4. 保留段落格式（粗体、斜体、上下标等）

### 方案 2：智能Run合并（次选）

**核心思路**：改进合并逻辑，忽略不影响翻译的格式差异

**优点**：
- ✅ 平衡了上下文和格式保留
- ✅ 减少大部分切分

**缺点**：
- ⚠️ 实现复杂
- ⚠️ 可能丢失某些格式信息
- ⚠️ 仍然会有一定程度的切分

**实现要点**：
`python
def should_merge(style1, style2):
    # 忽略这些格式差异
    ignore_keys = {'rFonts', 'lang', 'webHidden', 'noProof'}
    
    # 保留这些关键格式
    key_formats = {'b', 'i', 'u', 'vertAlign', 'color', 'sz'}
    
    relevant_style1 = {k: v for k, v in style1.items() if k in key_formats}
    relevant_style2 = {k: v for k, v in style2.items() if k in key_formats}
    
    return relevant_style1 == relevant_style2
`

### 方案 3：语义单元切分（复杂但最优）

**核心思路**：基于语义边界（句子、小节）而非格式边界切分

**优点**：
- ✅ 翻译质量和上下文最佳平衡
- ✅ 避免超长文本问题

**缺点**：
- ❌ 实现非常复杂
- ❌ 需要 NLP 分词和句子边界检测

## 测试文件夹未考虑到的问题

基于对文档的深入分析，以下是可能被忽略的问题：

1. **超链接（Hyperlink）**
   - 如果文本包含超链接，链接文本可能在 w:hyperlink 标签内
   - 当前切分可能破坏超链接结构

2. **书签和交叉引用**
   - w:bookmarkStart/End 标记可能打断连续文本
   - 交叉引用（如"见第X章"）需要保持完整

3. **脚注和尾注**
   - 脚注标记是独立的 run，会被切分
   - 脚注内容在 word/footnotes.xml 中，需要单独处理

4. **域代码（Field Codes）**
   - 目录、页码、日期等动态内容
   - w:fldSimple 或 w:fldChar 标记
   - 可能需要跳过翻译

5. **修订标记（Track Changes）**
   - w:ins, w:del 等修订标记
   - 需要决定是否保留修订痕迹

6. **注释（Comments）**
   - 批注内容在 word/comments.xml
   - 是否需要翻译注释？

7. **嵌入对象**
   - 图表、公式（MathML）、SmartArt
   - 图表标题和轴标签可能需要翻译

8. **样式和格式继承**
   - 段落样式可能影响文本外观
   - 字符样式可能被切分逻辑破坏

这些问题在当前测试文档中可能存在，建议进一步检查。
